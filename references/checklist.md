# 通用自检清单（任何项目交付前跑一遍）

逐条核查，每条回答：✅ 符合 / ❌ 不符合 / ➖ 不适用，并给出依据（文件或代码位置）和不符合时的风险。只诊断，先不要改。最后按“安全/资金 → 稳定性 → 结构规范”排序，标出最该先修的 3 项。

## 【安全与资金】

1. 前端是否未直接访问数据库（用 BaaS 时是否已正确配置行级权限）？
2. 关键权限判断是否在后端独立做（不只前端）？
3. 密码是否用 bcrypt/argon2/scrypt + 随机盐存储（非明文、非 MD5/SHA 自拼）？
4. 金额字段是否用整数最小单位或精确十进制（非 FLOAT/DOUBLE/REAL）？
5. 身份证/证件等是否加密存储 + 展示脱敏？支付卡信息是否交给持牌机构而非自存？
6. 不可删数据（订单/付款/合同/财务）是否软删除 + 审计？
7. 敏感配置（密码/API Key/密钥）是否未提交进 Git、未放进前端变量？
8. 默认账号是否强制改密、未在公开文档明文写死？
9. SQL 是否全部参数化（无字符串拼接）？
10. 关键接口是否按 user_id/租户过滤，防越权？登录等接口是否有限流？

## 【稳定性（防“一改就崩”）】

11. 是否有 Git 且能一键回退？真实数据是否有单独备份？
12. 是否有“红绿灯”（至少一条冒烟检查，核心功能有基础测试）？
13. 数据库建表脚本与代码引用字段是否完全一致（无靠静默 ALTER 维持的隐性不一致）？
14. 是否无“捕获异常后既不处理也不记录”的静默吞错？
15. 数据库结构是否有正规迁移文件作为唯一来源？

## 【结构与规范】

16. 目录、模块边界是否清晰？
17. 是否有设计 Token，样式是否未硬编码？
18. 同类 UI 是否已抽象复用（无大量复制粘贴）？
19. 文案是否统一管理（为多语言预留）？
