# Vibe Coding 项目实施与检查手册（通用版·修订版）

> 什么是 Vibe Coding：用自然语言指挥 AI Agent（Claude Code、Codex、Cursor 等）开发，你专注"要做什么"，把"怎么写"交给 AI。本手册为这种工作方式而写。

**适用对象**：用 AI Agent 开发、但不具备专业编程背景的人。
**怎么用**：全文每个动手处都给 🟦 可直接粘贴给 Agent 的指令。你只需：复制 → 粘贴给你的 Agent → 看它给你的"人话结论 + 红灯/绿灯"。

**符号图例（先认一下）**：
- 🟦 可直接复制粘贴给 AI 的指令。**看不懂没关系，照抄粘贴即可。**
- 🟢 新项目专用步骤
- 🟡 老项目专用步骤
- ⛔ 红线 / 铁律，不能妥协
- 💡 作者的强观点（有依据但非绝对真理），按项目情况取舍
- ✅ / ❌ / ➖ 自检结论：符合 / 不符合 / 不适用

**手册分三层**：
- 通用层：红线 + 老项目大检查 + 三阶段方法论 + 修复纪律。所有项目通用。
- 场景适配层：按 5 类项目给侧重点，对号入座。
- 自检清单层：一条条"是否"检查项，任何项目自己跑一遍。

**两种模式**：🟢 新项目 = 立规矩打基础（阶段一 → 二 → 三）；🟡 老项目 = 先做第 0 部分，再补齐。

> **名词约定**：本手册多次提到"真源文档"，指你项目里那份"唯一说了算"的说明文档（建议命名 `CLAUDE.md` 或 `AGENTS.md`，放在项目根目录）。所有定下来的规矩、选型、红线都写进它；每开一个新对话，先让 AI 读它。

## 与 AI 协作的四条铁律（先记住这个）

不懂代码的人最容易踩的，往往不是技术坑，而是"放任 AI 自由发挥"。这四条贯穿全程：

1. **删文件、删数据、改数据库结构、装新依赖——做之前必须先问你。** 🟦 把这句写进真源文档："涉及删除文件或数据、修改数据库结构、新增第三方依赖时，先停下来告诉我影响，等我同意再做。"
2. **同一个问题修 3 次还没好，就喊停。** 别让 AI 反复瞎猜。🟦"你已经试了 3 次都没成功，请先停止改代码，分析根本原因，列出 2 种不同思路让我选。"
3. **让它演示真实流程，而不是只说"我修好了"。** 🟦"请用真实操作（或写一个能跑的测试）证明这个功能现在确实正常，把过程贴给我看，别只说'已修复'。"
4. **重要规矩写进真源文档，每次开新对话先读它。** AI 不记得上次的约定。开新会话第一句：🟦"请先读项目根目录的 CLAUDE.md，严格按里面的规矩来。"

## ⛔ 通用红线（任何项目，先读这一条）

违反以下任意一条都应作为最高优先级处理。括号是"为什么"。

1. **前端不碰数据库、不存敏感数据。** 所有数据读写、校验、权限判断都经过后端。（前端代码用户可见可改，直连数据库 = 把钥匙交给所有人）
	- 例外说明：若你用 Supabase、Firebase 这类 BaaS（后端即服务），前端"看起来"在直连数据库，但前提是你已开启并正确配置了行级权限（RLS）等安全规则——本质仍是"服务端在把关"，不是裸连。没配规则就直连，等同于把库公开。
2. **"能不能做"的判断，后端必须独立再做一次。** 前端的隐藏、禁用只为体验，可被绕过，不是安全防线。（只在前端拦，等于没拦）
3. **密码绝不存明文，也绝不自己发明加密。** 用专门为密码设计的哈希算法（**bcrypt、argon2、scrypt** 三选一）加盐存储；**禁止**用 MD5、SHA1、SHA256 自己拼个盐就当密码存——它们是给文件校验用的，扛不住暴力破解。（数据库一旦泄露，弱哈希 ≈ 明文）
4. **钱绝不用浮点数。** 金额用"最小货币单位的整数"（人民币就用"分"），或精确十进制类型（DECIMAL / NUMERIC）；禁用 FLOAT / DOUBLE / REAL，也别直接用 JS 的 number 存金额入库。**用 SQLite 时它没有真正的 DECIMAL，请直接用整数存"分"。**（浮点有精度误差，对账、税额、累加迟早算错）
5. **个人敏感信息：能不自存就不自存；必须自存就加密 + 脱敏。**
	- 支付相关（银行卡号、CVV、支付密码）**不要自己存**，交给持牌支付机构（Stripe、微信支付、支付宝），你只保存它们返回的凭证号 / 订单号。
	- 确需保存的身份证、证件号等，**加密存储**（用成熟加密库；密钥放环境变量或密钥管理服务，绝不和数据写在一起、绝不进 Git），**展示时打码**。（本地数据库文件被拷走，明文 = 全部泄露）
6. **不可篡改删除的数据用软删除 + 审计。** 订单、付款、合同、财务记录只标记删除、保留"谁在何时做了什么"。
7. **敏感配置不进 Git。** 数据库密码、API Key、密钥放环境变量（`.env` 文件）或被 `.gitignore` 排除的本地配置，绝不提交到仓库。**注意：放进前端的密钥等于公开**——凡是带 `NEXT_PUBLIC_`、`VITE_` 前缀的变量都会被打包进浏览器能看到的代码里，绝不能放真正的密钥。
8. **默认账号口令不可写死且公开。** 若有初始管理员，必须强制首次改密，不在公开文档里明文写死。

> 💡 作者强观点：涉及"钱"或"真实身份信息"的项目，以上 8 条没有"看情况"，一条都不能省。体验、动画、好看，永远排在它们后面。

## 第 0 部分 · 老项目大检查（🟡 存量项目从这里开始）

专治"本意是优化，结果一改就崩"。根因几乎都不是你改错，而是改前没有安全网、改后无法验证。本部分让 Agent 替你建好安全网，你只管复制指令、看结论。

### 0.1 为什么会"一改就崩"（先懂道理）
三个原因：① 改动范围失控（想改 A，Agent 顺手动了 B C D）；② 没有验证手段（改完只能肉眼打开看，看不到的地方坏了不知道）；③ 没有存档（改坏退不回）。下面三步正好各治一个。做完这三步前，别让 Agent 大规模优化老项目。

### 0.2 第一步：让 Agent 建好"后悔药"（存档）
🟦 复制给 Agent：
> "请检查本项目是否用 Git 管理。若没有，帮我初始化并完整提交一次，说明写'大检查前的原始版本'。然后用大白话告诉我：以后每次想改东西前，我该对你说哪一句话，能保证万一改坏可一键退回。"

> ⚠️ **Git 存的是"代码"，不是"你的数据"。** 数据库里的真实数据（用户、订单等）并不在 Git 里。重要数据要单独定期备份——让 Agent 给你一条导出/备份命令，并说清楚备份存到哪、怎么恢复。

### 0.3 第二步：让 Agent 装好"红绿灯"（自动验证，但量力而行）
⚠️ **重要顺序**：对老项目，一上来就强加严格的类型检查、Lint 和全面测试，往往会冒出成百上千个报错，把你吓退、甚至诱导 AI 大改一通。正确做法是"先有一个能用的冒烟检查，再逐步加严"。
🟦 复制给 Agent：
> "我不懂代码，需要一个能快速告诉我'项目有没有坏'的机制，但请循序渐进，不要一上来就改一大片：① 先给我一条最简单的'冒烟命令'——项目能启动、关键页面能打开、登录等核心功能不报错，就算绿灯；② 再检查是否已有类型检查、代码规范检查（Lint）、自动化测试；缺的先别全开，只给登录、权限校验、涉及金额或敏感数据的核心功能补最基本的测试；③ 给我一条命令，运行后你告诉我现在是'全部正常（绿）'还是'哪里坏了（红）'；④ 用三行以内大白话说明：怎么运行、看到什么算好、看到什么算坏。注意：纯展示型前端页面不必强写测试，重点放在'核心功能不崩'。"

> 💡 作者强观点：哪怕是本地桌面单机工具，也值得有这条"红绿灯"。很多人以为单机软件不用测试，恰恰是这类项目最常"换台电脑就打不开"。

### 0.4 第三步：让 Agent 只"体检"不"动手术"（先诊断）
⚠️ 关键：先只出报告，绝不让它改代码。
🟦 复制给 Agent：
> "请只做诊断，不要改任何代码。按严重程度排序（安全/资金 → 数据 → 功能 → 体验）输出问题清单，逐项检查：① 前端是否直接连数据库、或关键判断只在前端做；② 密码是否明文或用了弱哈希（MD5/SHA 自拼盐）；金额字段是否误用了浮点类型（REAL/FLOAT/DOUBLE 或直接用浮点数存钱）；③ 身份证、银行卡、证件等敏感信息是否明文存储、展示是否脱敏；支付卡信息是否在自己库里乱存（应交给持牌机构）；④ 数据库实际结构与代码引用的字段是否一致，是否存在靠静默补丁（反复 ALTER、try/catch 吞错）维持的隐性不一致；⑤ 是否存在静默吞掉错误的代码（捕获异常后既不处理也不记录），导致出问题无感知；⑥ 敏感配置（密码、API Key、密钥）是否被提交进了 Git，或被放进了前端能看到的变量里；⑦ 目录、组件复用、样式是否混乱；数据库结构是否有正规迁移文件作为唯一来源；⑧ 缺哪些基础保障（Git、冒烟检查、类型检查、Lint、测试、数据备份）。每项说明：在哪、风险是什么、建议怎么修、不修会有什么后果。先别动手。"

### 0.5 第四步：小步修复纪律（从此不再崩）
🟦 每次修复都这样开头（复制并填空）：
> "我们现在只修这一个问题：【粘贴报告里的某一条】。要求：只改与这个问题直接相关的地方，不要动其他任何功能，尤其不要顺手重构、改样式或动画。改完自己运行体检命令确认绿灯，再用一句话告诉我改了什么。若发现必须连带改别处，先停下告诉我影响范围，等我同意再继续。"

修复优先级（严格按此顺序）：**通用红线 → 资金/数据正确性 → 高耦合"地雷" → 功能 bug → 体验/样式/动画放最后。**

> ⛔ 铁律："每次只改一小步，改前先存档，改后看绿灯，坏了就反悔。安全、资金、数据永远排在体验前面。"

## 开工前 · 把地基环境对齐（🟢 新项目第 0 步）

很多"AI 装完跑不起来""换台电脑就报错"，根子在环境没对齐。先花两分钟做这件事，能省后面大量麻烦。
🟦 复制给 Agent：
> "在开始写任何代码前，请先：① 探活当前环境，告诉我已经装了什么（语言版本、包管理器等）、还缺什么、要不要我装；② 确定并固定一个包管理器（比如 npm、pnpm、yarn 只选一个），写进真源文档，以后统一用它，别混用；③ 把项目要求的语言/运行时版本也记下来。用大白话告诉我结论，缺东西就给我安装指令。"

## 第一阶段 · 前端骨架（🟢 新项目从这里开始）

让 Agent 一股脑横推所有页面，是项目越写越乱的根源。先把前端骨架定下来，让所有页面"长在同一套规则上"。

> 🟡 老项目：若"每个页面风格都不一样、按钮弹窗不统一"，就是缺本阶段骨架。先用 0.4 体检报告定位，再逐步补规矩，切忌一次性重构。

### 1.1 先定整体设计风格
设计风格是整个界面的统一视觉语言。常见风格：玻璃拟态、Material Design、企业级管理台、现代极简、科技感（深色发光）、信息流。最终都会落到：颜色、字体、字号、层级、行高、间距、圆角、卡片、按钮形态、布局、信息密度。
🟦 复制给 Agent：
> "我的项目是【填项目类型】。请推荐 1 套最适合的整体设计风格，说明它在颜色、字号、间距、圆角、卡片、按钮、信息密度上的具体规范，以及为什么适合我的项目。先不要写页面代码。"
> 不确定喜欢什么时补一句："我发你几个我喜欢的网站链接，请分析它们的视觉风格和布局，判断适不适合我的项目。"（目的是定方向，不是照抄）

### 1.2 选前端技术方案与 UI 组件库（两者分开）
- 前端技术方案管"项目怎么组织和运行"：Web 常见 React、Vue、Angular，及配套 Next.js（React）、Nuxt（Vue）；小程序 / 跨端有微信小程序原生、Taro、uni-app；桌面端 Electron、Tauri。
- UI 组件库管"界面元素怎么统一"：Ant Design、Element Plus、Naive UI、MUI、Mantine、Vant（移动端）；特殊方案 Radix UI（无样式基础库）、shadcn/ui（组件源码方案）。
- Tailwind CSS 是原子化 CSS 框架，不是组件库；基于它的组件方案有 DaisyUI、shadcn/ui。
> 一句话：框架管项目结构，组件库管界面统一。
> 💡 作者强观点：不懂技术、主要靠 Agent 写时，优先选成熟、文档全、社区活跃、AI 熟悉的方案，别为"显得高级"选冷门库。技术路线一旦摇摆，项目很快就乱。
🟦 复制给 Agent：
> "请根据我的项目类型和已定设计风格，各推荐一个最适合的前端框架和 UI 组件库，分别说明为什么选它、为什么不选其他。优先成熟、文档完整、社区活跃、AI 熟悉度高的方案，并指明使用它们的**最新稳定版本**。另外请帮我确认：推荐的库目前是否仍在活跃维护、开源协议是否允许我的用途（避开 GPL 这类有传染性的协议，除非我清楚后果）。确定后写进真源文档，作为以后必须遵守的标准。先不要写页面。"

### 1.3 定目录、模块、组件复用规则
🟦 复制给 Agent：
> "写任何页面前，请先建立并写进文档：① 目录规范（页面、组件、接口请求、工具函数、样式各放哪）；② 模块边界（各功能区域分开）；③ 组件复用铁律：同类 UI 结构出现超过 2 次，必须抽象成可复用组件，禁止复制粘贴。能用组件库就用，不够再基于组件库封装业务组件。"

### 1.4 提前准备样式系统
🟦 复制给 Agent：
> "请建立统一样式系统并写进文档、设为强制规范：① 设计 Token（颜色、字号、行高、间距、圆角、阴影统一定义，所有样式必须从 Token 取，禁止页面里硬编码任意值）；② 多语言文案统一管理、不硬编码（考虑中英文长度差异，避免按钮撑爆、标题换行）；③ 主题规则：先定默认主色，预留主题扩展，主题色集中管理。"

### 1.5 写进实施计划，再搭骨架
🟦 复制给 Agent：
> "请把以上决定整理成实施计划文档，列出第一阶段范围和验收标准。然后只搭骨架，不做全部页面。验收：项目能启动、首页能打开、路由清楚、目录规范、组件库接入、Token 与主题就绪、常用组件有基础版本。完成后运行体检命令确认绿灯，再提交 Git。"
> 本阶段不追求做多少页面，只追求"基础结构对、能跑起来"。跑通并提交后，再一个模块一个模块加功能。

## 第二阶段 · 数据库设计

> 💡 作者强观点：建议前端主流程成型后再设计数据库——此时更清楚有哪些业务对象，Agent 才不会凭空猜表。（若你更习惯先设计数据，也可，但务必先把业务对象和流程讲清楚。）数据库一旦乱，前端、权限、交易、统计都会被拖累，数据量大后还会变慢。正确做法不是慢了再升级服务器，而是一开始就设计清楚。
> 🟡 老项目：若"字段乱、表关系看不懂、接口对不上"，对应本阶段。⚠️ 数据库改动风险最高，任何改表必须先存档、先出迁移文件、再执行（见 2.6）。

### 2.1 从业务里找对象
🟦 复制给 Agent：
> "请先不要建表。根据我的页面、用户流程和核心功能，整理出有哪些业务对象、各自对应什么业务场景。前端已完成的可按页面反推数据。信息不全就主动问我，不要漏掉对象。"

### 2.2 确认对象之间的关系
🟦 复制给 Agent：
> "请说明每个对象对应一张什么表，表间是一对一/一对多/多对多，关联字段放哪张表，为什么这样设计。请确认你理解的是我的业务，而不是自己编的结构。"

### 2.3 选数据库
- 💡 关系型（MySQL / PostgreSQL）：有用户、内容、权限、交易记录的项目优先，更稳。
- SQLite：本地小工具、桌面单机应用、轻量存储。（多人同时写入、需远程访问时会力不从心）
- Redis：不是主库，用于缓存、登录态、验证码、限流、队列。无明确高并发/缓存需求时，别一开始就上。
🟦 复制给 Agent：
> "请推荐一个主数据库并说明原因；判断我的项目当前是否真需要 Redis，不需要就明说；说明本地开发和线上部署分别怎么处理。优先成熟、文档多、AI 熟悉的方案。"

### 2.4 设计表结构与字段（三条大白话原则）
理解作用即可，不必记术语：
1. **一格只放一件事**：一个字段只存一个含义，别把"姓名 + 电话"塞进同一个字段。
2. **每张表只管自己的主角**：订单表放订单信息，用一个"关联字段"——比如存 `user_id`（用户编号），而不是把用户的姓名、电话整套复制进来——连到用户表。
3. **能"查得到"的就别"抄一遍"**：能通过关联查到的信息，别在多张表里重复存；否则改一处、忘改另一处，数据就对不上。
> 两个合理例外，都须 Agent 先说明原因：**快照**（如订单保留下单时的收货信息，即使用户之后改了地址也不影响历史订单）；**为性能有意冗余**。

### 2.5 提前规范常见字段 + 红线
🟦 复制给 Agent：
> "设计表时遵守并写进文档：① 字段用英文 + 下划线命名；每张表含主键 `id`、`created_at`、`updated_at`；② 不可物理删除的数据加 `deleted_at` 软删除，保留审计；③ 密码只存强哈希（bcrypt/argon2/scrypt）+ 每用户随机盐，绝不明文、绝不用 MD5/SHA 自拼；④ 金额绝不用浮点（FLOAT/DOUBLE/REAL），用最小货币单位整数或精确十进制类型，明确币种和小数位（SQLite 用整数存'分'）；⑤ 身份证/证件等敏感字段加密存储、展示脱敏，支付卡信息不自存、交给持牌机构；⑥ 状态字段用明确枚举；⑦ 关键业务字段类型先问我确认再定。"

### 2.6 先出设计文件，再建表（数据库改动铁律）
🟦 复制给 Agent：
> "不要直接连库建表。先生成数据库设计文件（`schema.sql` 或框架的迁移文件），让我核对对象、关系、字段，确认后再执行建表，并提交 Git——它是数据库结构的唯一来源。以后任何改表，必须先改迁移文件、说明原因，再执行；禁止直接改库，也禁止用'反复 ALTER + 静默 try/catch'偷偷补结构。"
> ⛔ 特别红线：代码里引用的字段，必须和数据库实际建出来的字段**完全一致**。若靠一堆"补丁式 ALTER"维持，换新电脑全新建库时极可能字段缺失、登录失败、白屏。务必让 Agent 核对"建表脚本"与"代码实际用到的字段"是否对得上。

### 2.7 写进实施计划再动手，只跑通基础结构
🟦 复制给 Agent：
> "请把业务对象、关系、主库选型、是否用 Redis、核心表及职责、字段规格、表间关联、哪些数据不可物理删除、是否符合上面三条设计原则、验收标准，写成数据库实施计划。然后只跑通基础结构：能连库、建核心表、创建并执行迁移、插入与查询测试数据即可。完成后跑体检命令、确认绿灯、提交 Git，再写接口。"

## 第三阶段 · 后端技术认知与骨架

后端是"用户看不见、但负责业务逻辑、数据处理、权限校验、接口调用、任务执行"的部分。分工：前端让用户操作并展示；数据库存数据；后端判断"这件事能不能做、该怎么做、做完后数据要不要写入或变更"。
> 🟡 老项目：若"接口很乱、前端接不上、数据库对不上"，对应本阶段。后端是红线集中地，优先核查"权限和敏感判断是否真在后端、SQL 是否参数化"。

### 3.1 先分清两种后端场景
- **小脚本**（整理文件、处理表格、定时任务）：一个脚本即可，不必硬上完整后端。
- **项目级后端**（用户系统、交易、内容、权限、支付回调、审核、统计）：管三件事——① 业务规则（谁能做什么）；② 数据流转（前端提交的数据先校验再处理，最后才写库）；③ 接口协作 API（前后端约定的交互入口）。
> ⛔ 红线：业务规则和权限判断必须在后端，前端隐藏/禁用只是体验，可被绕过。

### 3.2 认识常见后端语言（了解适用场景即可，别纠结）
- **Python**：上手简单，适合脚本、自动化、数据处理、AI 应用，也可做轻量后端。
- **Node.js**：前端已用 JS/TS 时，前后端语言统一、理解成本低。
- **Java / Go / C#（.NET）/ PHP**：都是成熟的后端方案，各有侧重——Java 偏企业级、Go 部署简单且性能好、C# 和 .NET 如今已是跨平台（不再只限 Windows）、PHP 生态成熟上手快。
> 不懂技术时不必纠结选哪个，让 Agent 按你的项目给**唯一主推荐**即可，别让它列一堆选择题。
🟦 复制给 Agent：
> "请根据我的项目类型、核心功能、用户规模、数据库方案、部署方式和后期维护难度，推荐一个最适合的后端语言。只给一个主推荐，不要列选择题，说明为什么选它、为什么不选其他，并指明使用其最新稳定版本。"
> 💡 作者强观点：最忌技术路线摇摆，让 Agent 给唯一主推荐。

### 3.3 框架比语言更影响项目质量
语言决定"用什么写"，框架决定"按什么规矩写"。框架分两类：轻量型（主要管路由和 API，规范要自己补）和完整型（自带项目结构、数据库支持、配置、权限、命令行、任务等）。
> 💡 作者强观点：权限复杂、涉及资金的项目，优先选完整型框架。轻量接口/内部小工具可用轻量框架，但仍须让 Agent 明确目录、数据库访问、参数校验、权限、错误返回。
🟦 复制给 Agent：
> "请基于已选后端语言，推荐一个适合我项目的框架。说明：适合什么场景、官方目录规范、哪些能力框架自带、哪些需额外 SDK/库。优先成熟、文档完整、社区活跃、AI 熟悉度高的方案，并指明最新稳定版本。"

### 3.4 先问业务，不要先问代码
🟦 复制给 Agent：
> "请基于我的前端流程、数据库设计和业务需求，先分析后端方案，不要写代码：① 后端负责哪些业务逻辑；② 前端需要哪些 API；③ 每个 API 涉及哪些表；④ 每个 API 的权限校验、参数校验和异常处理；⑤ 哪些接口需登录、哪些需管理员；⑥ 身份认证怎么做、权限校验放哪一层、错误返回格式如何统一。"

### 3.5 后端安全基线（涉及资金/隐私的项目必须加做）
🟦 复制给 Agent：
> "若我的系统处理金额或个人敏感数据，请把以下设为后端强制基线并写进文档：① 敏感操作和权限判断在后端独立校验；② 所有外部输入在后端做参数校验；③ 数据库操作使用参数化查询，严禁字符串拼接 SQL；④ 密码用强哈希（bcrypt/argon2/scrypt）存储；⑤ 身份证/证件加密存储、展示脱敏，支付卡信息交持牌机构、不自存；⑥ 资金和合同的关键操作记录审计日志，数据只软删除；⑦ 敏感配置放环境变量，绝不进 Git、绝不放进前端变量；⑧ 错误必须被捕获并记录日志，禁止'捕获后既不处理也不记录'的静默吞错；⑨ 日志里不得打印明文密码、身份证、卡号、令牌（token），必须脱敏；⑩ 登录、发短信验证码、找回密码等接口加限流，防恶意刷。"

### 3.6 写进真源文档，防止中途换路线
🟦 复制给 Agent：
> "请把本次后端语言、框架、API 边界、业务规则、安全基线写进项目真源文档，后续开发以此为准。如需更换语言、框架或核心 SDK，必须先说明原因并更新文档，不得中途随意换路线。"

### 3.7 搭骨架，只跑通基础结构
🟦 复制给 Agent：
> "请把上述决定整理成后端实施计划（含第一阶段范围和验收标准），然后只搭后端骨架，不写完所有接口。验收：能启动、目录规范、数据库能连、有统一错误返回和基础认证/权限框架、有 1~2 个示例接口跑通。完成后运行体检命令确认绿灯，再提交 Git。之后一个接口一个接口加，每个都先想清业务规则和权限再写。"

### 3.8 上线部署（简版，能用就行）
不懂运维也能上线，关键是别把"本机能跑"当成"别人能用"。
🟦 复制给 Agent：
> "我要把项目上线给别人用，但我不懂运维。请：① 根据我的技术栈，推荐一个对新手最友好的部署方式（如前端用 Vercel、数据库/后端用 Supabase、Render 或 Zeabur 等），说明为什么；② 一步步教我怎么部署，把要点操作列成清单；③ **务必启用 HTTPS**（让网站地址是 https 开头），并解释为什么没有它登录和支付都不安全；④ 告诉我线上环境变量/密钥该在哪里配置，确保它们不进 Git、不进前端。"

## 当 AI 意见冲突时怎么办（方法论核心）

> 换个 Agent 或模型审查，就给出不同甚至相反的意见。这不代表谁错了。关键是分辨"这是哪一类分歧"，再用对应方法处理。

### 三类分歧，分开对待
1. **客观对错**（有唯一答案，可验证）：如金额用浮点有误差、明文存密码不安全、SQL 拼接有注入风险。→ 不存在合理的不同意见，能用事实/测试验证，照标准做。
2. **工程权衡**（没有唯一答案，看场景）：如用关系型还是 NoSQL、要不要上 Redis。→ 本来就允许多种答案，都可能对，取决于项目规模与未来方向。
3. **风格偏好**（无所谓对错）：如命名、目录怎么分。→ 选一个、保持一致即可。
> ⚠️ 最常见错误：把第 2、3 类的偏好差异，误当成第 1 类的"我错了"，于是反复推翻、越改越乱。

### 破局原则：别问"该信谁"，改问"能不能验证"
- 能验证 → 让它拿证据说话。
- 不能验证（权衡/偏好）→ 没有标准答案。选一个有依据、且与项目其余部分一致的方案，写进真源文档锁定，不再摇摆。
> ⛔ 对非技术用户，"反复摇摆"的危害远大于"选得不够完美"。锁定比纠结更重要。

### 裁决提示词（遇到分歧时用——看不懂直接复制）
🟦 复制给 Agent：
> "针对【某个具体分歧】，请不要只给结论。请告诉我：① 这是有客观对错、看场景权衡、还是纯风格偏好？② 若有客观对错，给出可验证依据（标准、文档、能跑的测试或例子）；③ 若是权衡，列出各方案的前提和代价，并基于我的项目实际给一个明确推荐；④ 若只是偏好，直接说'无所谓对错，选一个保持一致即可'。不要用'最佳实践''业界通用'这类空话，要给我能验证或能权衡的具体内容。"
> 本质：把"比谁更权威"变成"比谁拿得出证据"。拿不出证据、只靠"这样更专业"压你的，无论哪个模型，都可降低权重。

## 场景适配层 · 按项目类型对号入座

> 通用层适用所有项目。这一层只讲各类项目额外要盯紧的点，找到你的类型看对应几条即可。

### 类型一 · 后台管理 / SaaS 工具
- **权限是命脉**：角色 × 资源 × 操作的权限矩阵在后端强制校验；侧边栏/按钮过滤只是体验，每个接口仍要独立鉴权。
- **防越权**：列表和详情接口必须按"当前用户的 `user_id` / 所属租户"过滤数据，否则用户改一下网址里的编号，就能看到别人的数据。
- **列表性能**：数据量大，必须分页 + 关键字段加索引。
- **审计日志**：涉及配置、权限、金额的操作要可追溯。
- 💡 优先完整型后端框架 + 成熟管理台组件库。

### 类型二 · 电商 / 交易 / 支付
- ⛔ **金额、库存、订单状态全是红线**：金额用整数/精确十进制；下单时校验库存和价格（不信前端传来的价格）；订单状态用枚举且按规则流转。
- ⛔ **防超卖**：扣库存、下单要用数据库事务 + 锁，避免两个人同时买最后一件都成功。
- ⛔ **支付回调必须验签 + 幂等**：同一回调可能重复到达，不能重复加钱/发货。
- **快照**：下单时保存当时价格、收货信息。
- **对账**：金额相关务必有审计和软删除。

### 类型三 · 内容平台 / 社区
- **权限与归属**：谁能编辑/删除谁的内容，后端校验归属。
- ⛔ **内容安全**：用户提交的内容在展示前要做转义，防 XSS（别让用户输入的脚本在别人页面里偷偷运行）；并做审核、防违规。
- **防刷**：注册、登录、发帖、点赞等接口加限流。
- **关系建模**：用户-文章-评论-点赞-收藏的多对多关系理清。
- **性能**：信息流列表分页，热点数据可用缓存（此时 Redis 才有意义）。

### 类型四 · 本地桌面 / 单机工具
- ⛔ **"本地存储 ≠ 安全"**：本地数据库文件可被拷走，敏感字段仍要加密。
- **数据与程序分离**：用户数据存独立、卸载不删除的路径；升级/重装不丢数据。
- **数据库结构一致性**：换电脑全新建库时，建表脚本必须和代码引用字段完全一致，否则典型表现就是"换电脑就白屏/登录不了"。
- 💡 即便单机，也建议建"红绿灯"和定期数据备份快照。

### 类型五 · 小程序 / App
- **后端是唯一可信方**：客户端代码完全暴露，所有校验、价格、权限判断必须在后端。
- **接口版本兼容**：用户不会立刻更新客户端，后端改接口要兼容旧版本。
- **密钥不进客户端**：第三方密钥放后端，客户端只拿临时凭证。
- **多端文案与适配**：文案统一管理，注意机型尺寸适配。

## 附：一个真实匿名案例（说明这套清单真的抓得出问题）

一个用 AI 协作半年、相当规范的桌面端管理类项目（本地数据库 + 独立后端 API），用上面清单体检：
**亮点（方向对）**：有成熟的 Agent 约定文档、密码加盐哈希、SQL 全参数化、权限系统、审计日志、路径遍历防护。比多数同类项目规范。
**但清单仍抓出关键雷**：
- ❌ 第 4 条：金额字段全用浮点类型，存在对账算不准隐患。
- ❌ 第 5 条：身份证、银行卡明文存储，且对外宣称"本地很安全"。
- ❌ 第 8 条：默认口令写死且公开在说明里。
- ❌ 第 13 条：建表脚本与代码字段对不上，靠"补丁式 ALTER + 静默 catch"维持，换新环境极易出问题。
- ❌ 第 14 条：多处静默吞错，出问题无感知。
**启示**：即便项目整体很规范，这些"看不见的雷"也只有靠系统性清单才能逐条揪出。本手册不替你写代码，而是让你心里有谱、改得安全。

## 一页纸速查（贴在最显眼处）

**🟢 新项目立规矩**（依次发给 Agent，每步跑绿灯 + 提交 Git）：
1. 环境对齐（探活 + 固定包管理器）→ 建 Git 存档 + 数据备份意识 → 建红绿灯（先冒烟，再逐步加严）→ 定前端风格与选型 → 定前端规矩（目录/复用/Token/多语言/主题）→ 搭前端骨架 → 设计数据库（找对象 → 定关系 → 选库 → 三条原则 + 红线 → 先出迁移文件 → 再建表）→ 规划后端（先分析不写码，定唯一语言/框架 + 安全基线）→ 搭后端骨架 → 部署上线（含 HTTPS）→ 逐模块迭代。

**🟡 老项目大检查**：
建存档（+ 数据备份）→ 建红绿灯（先冒烟）→ 只诊断不改（跑通用自检清单）→ 按"安全/资金 → 稳定性 → 结构"小步修复。

**⛔ 全程铁律**：
"每次只改一小步，改前先存档，改后看绿灯，坏了就反悔。安全、资金、数据永远排在体验前面。"

**🤝 与 AI 协作四铁律**：
删/改/装依赖先问我 · 修 3 次不成就停 · 让它演示真实流程 · 规矩写进真源文档、开新对话先读它。

**⛔ 八条红线**：
前端不碰库 · 后端独立鉴权 · 密码强哈希(bcrypt/argon2/scrypt) · 金额非浮点 · 敏感信息加密脱敏(支付卡交持牌机构) · 关键数据软删除+审计 · 密钥不进 Git/不进前端 · 默认口令强制改。
